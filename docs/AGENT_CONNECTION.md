# The full agent pipeline — how everything is connected

```
news_agent (:8002)                          Sanctions_agent
  scanner (mock news / NewsAPI)               watchlist/monitor.py
  ER + triage (Groq, keyless fallback)        screens the customer book
  emitter                                     (near-real-time delta stream)
     │ POST /signals/ingest                      │ POST /api/ingest
     ▼                                           ▼
        pipeline service — investigation_agent (:8001), ONE process
          1. ambiguity: resolve entity vs customer book (ER ladder, Rungs 0-3)
             CONFIRMED | SUPPRESSED (with plain-language reason) | AMBIGUOUS
          2. assess -> tier (golden fixtures until core/scoring lands)
          3. investigate: plan/execute/adjudicate (LLM or deterministic)
             evidence chain CONFIRMED / CORRELATED / MISSING
          4. draft SAR
          5. persist Case (+customer +candidates +assessments embedded),
             SAR, append-only AuditEvents  ->  investigation_agent/ckyc.db
     │
     ▼
wkyc/api/server.py (:8787) — read-API adapter
  projects contracts.models blobs -> the six-screen JSON the Flutter app parses
     │
     ▼
Flutter dashboard (lib/) — ApiRepository, unchanged
```

## Run it

```bash
./run_backend.sh          # starts everything, streams both producers, leaves it serving
flutter run -d chrome     # in another terminal
```

One-time setup (venv per agent) is documented at the top of `run_backend.sh`.

## The shared dataset

`investigation_agent/fixtures/customers.json` is THE customer book. The news agent
seeds its watchlist from it (`SHARED_CUSTOMERS_JSON` env to override); the ambiguity
resolver adjudicates against it; mock news (`news_agent/data/mock_news.json`,
regenerated by `scripts/generate_mock_news_shared.py`) covers its names. Duplicate
names ("Dipak Dwiwedi" ×2) collapse to one watched name — the ambiguity agent is what
splits them back into distinct clients, and that split IS the demo:

| what happens | result |
|---|---|
| SEBI debars "Dipak Dwiwedi" (PAN ATDPD…) | C1001 CONFIRMED → HIGH case + SAR |
| same name, customer C1002 (PAN AFOPG…) | SUPPRESSED — `PAN_MISMATCH_REJECT`, reason attached |
| UAPA addition "Hafiz Muhammad Saeed" | C1007 CONFIRMED → CRITICAL, investigation runs, SAR |
| news: NIA terror probe names Saeed | same case, timeline ACCUMULATES the signal |
| news: "Amir Khan announces new film" | triaged benign — never emitted |

## Connection points (what was added where)

| Where | What |
|---|---|
| `investigation_agent/api/ingest.py` | adapters: news Signal JSON → contracts Signal; sanctions trigger → contracts Signal; name→customer resolution over the shared book |
| `investigation_agent/api/main.py` | `POST /api/verify`, `POST /api/ingest` (sanctions), `POST /signals/ingest` (news) |
| `investigation_agent/core/verify.py` | `verify_hit()` — ER-ladder verdict summary |
| `investigation_agent/core/orchestrator.py` | `run_pipeline(external_signals=, extra_watchlist=)`; SIGNAL timeline events + `SIGNAL_INGESTED` audit rows; `_merge_with_prior` so re-ingests ACCUMULATE the timeline instead of clobbering it |
| `investigation_agent/db/store.py` | `persist(customer=, candidates=)` — embeds both into the Case blob for the UI |
| `Sanctions_agent/watchlist/monitor.py` | streams sanction hits → `/api/ingest` (`--verify-only`, `--dry-run`) |
| `news_agent/signals/scanner.py` | actually calls `emitter.emit()` (it never did) + audit row |
| `news_agent/signals/kyc_list.py` | `seed_from_shared_dataset()` replaces the hard-coded demo trio |
| `news_agent/signals/triage_agent.py` | keyless deterministic ER/triage fallback (Groq when key set) |
| `news_agent/signals/news_fetcher.py` | fix: mock articles now carry their content_hash (dedup collapsed everything to one article) |
| `news_agent/signals/.env` | `USE_MOCK_NEWS=true`, `CORE_API_URL=…:8001/signals/ingest` |
| `wkyc/api/server.py` | read-API adapter over the pipeline sink → six-screen JSON |
| `wkyc/run_backend.sh` | one-command full stack |

## Verified end-to-end (single `./run_backend.sh` run)

- Sanctions: 6 hits → 4 CONFIRMED, 2 SUPPRESSED (PAN mismatch, reason attached).
- News: 8 articles → 4 adverse emitted (terror ×2, SEBI debar, bribery), 4 benign
  suppressed at triage ("appointed to board" is NOT adverse).
- CASE-C1007 timeline: sanctions signal → CRITICAL reassessment → SAR → two news
  signals, all on one accumulated timeline.
- Read API serves alerts / entity360 / timeline / case / sar / audit / suppressions /
  metrics in the exact shapes `lib/models/models.dart` parses. `flutter analyze` clean.
- Pipeline test suite: 22/22 pass.

## Notes / next

- **Reviewer write-back** (approve/deny → flip Case status + append audit row in one
  transaction) is the remaining loop — the UI phase item.
- `core/scoring.py` is still the golden-fixture stub; when Vaibhav ships `assess()`,
  tiers respond to live signals instead of fixtures. The seam is already wired.
- With a `GROQ_API_KEY` in `news_agent/signals/.env` the news ER/triage goes LLM;
  with `ANTHROPIC_API_KEY` the investigation adjudicator goes LLM. Keyless runs use
  the deterministic fallbacks end-to-end.
- The old `wkyc/db/seed.py` demo sink is retired: the adapter reads the PIPELINE's
  `investigation_agent/ckyc.db` (override with `CKYC_DB=`).
